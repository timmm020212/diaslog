# Diaslog Spy bot — Спецификация (дизайн)

Дата: 2026-06-13

## Цель
Локальная Python-программа для **личного** аккаунта пользователя, которая ловит в
**личных переписках** (1-на-1):
- удалённые сообщения (текст + медиа),
- изменённые (отредактированные) сообщения,
- одноразовые (view-once) фото и видео,

и присылает их пользователю в личку через отдельного бота (как «Dialog Spy bot»).

## Ограничения и предпосылки
- Только **личные** чаты. Группы и каналы НЕ отслеживаются.
- Работает только на **своём** аккаунте пользователя (его переписки).
- Требует userbot-доступ (MTProto), т.к. обычный Bot API не видит удалённые/правленые
  сообщения и view-once медиа.
- ⚠️ Автоматизация аккаунта формально против ToS Telegram — риск блокировки минимален при
  аккуратном личном использовании, но существует.
- Программа должна работать постоянно (24/7). На первом этапе — локально на ПК (Windows).
  При выключенном ПК бот не ловит события.

## Архитектура
Один процесс на библиотеке **Telethon**, две части:
1. **Юзербот** — `TelegramClient`, логин по номеру телефона (api_id/api_hash с my.telegram.org).
   Слушает события личных чатов, ведёт кэш, ловит удаления/правки/view-once.
2. **Бот-доставщик** — `TelegramClient.start(bot_token=...)`, токен от @BotFather.
   Отправляет пользователю оформленные уведомления в личку.

Связь между частями — внутри процесса (юзербот вызывает методы доставки бота).

## Поток данных
1. **Входящее личное сообщение** → обработчик:
   - сохраняет запись в SQLite (chat_id, msg_id, sender, text, тип медиа, timestamp);
   - если включено кэширование медиа — скачивает медиа в локальную папку;
   - если это **view-once** медиа — немедленно скачивает и доставляет пользователю (👁).
2. **Событие удаления** (`events.MessageDeleted`) → ищем msg_id в кэше:
   - найдено → доставляем: отправитель, текст, и медиа (если было закэшировано) — 🗑;
   - не найдено (пришло до старта программы) → тихо пропускаем или короткая пометка.
3. **Событие редактирования** (`events.MessageEdited`) → сравниваем с кэшем:
   - доставляем «было … → стало …» (✏️), затем обновляем кэш.

## Компоненты (файлы)
- `main.py` — точка входа: инициализация обоих клиентов, регистрация обработчиков, запуск.
- `cache.py` — слой хранения: SQLite (метаданные сообщений) + папка медиафайлов;
  автоочистка записей и файлов старше 7 дней.
- `delivery.py` — форматирование уведомлений и отправка через бот-клиент.
- `handlers.py` — обработчики событий (new / deleted / edited / view-once). (может быть в main.py)
- `config.py` + `.env` — секреты и настройки (api_id, api_hash, bot_token, owner_id,
  cache_media on/off, retention_days).
- `requirements.txt` — зависимости (telethon, python-dotenv).
- `README.md` — пошаговая инструкция настройки и запуска.

## Обработка ошибок / краевые случаи
- Удаление сообщения, которого нет в кэше → пропуск (или краткая пометка «содержимое не сохранено»).
- Ошибка скачивания медиа → доставляем текстовую пометку без файла.
- Бот не может писать пользователю (не нажат «Старт») → лог с инструкцией.
- Разрыв соединения → Telethon переподключается автоматически.
- View-once «сгорел» до скачивания → лог, пропуск.

## Настройки (config / .env)
- `API_ID`, `API_HASH` — с my.telegram.org.
- `BOT_TOKEN` — от @BotFather.
- `OWNER_ID` — Telegram user id получателя уведомлений (сам пользователь).
- `CACHE_MEDIA` — кэшировать ли входящие медиа для восстановления удалённых (по умолч. вкл).
- `RETENTION_DAYS` — сколько хранить кэш (по умолч. 7).

## Тестирование
- Юнит-тесты для чистой логики: `cache.py` (запись/поиск/очистка), форматирование в `delivery.py`
  на фейковых данных.
- MTProto-события тестируются вручную: со второго аккаунта прислать ЛС → удалить/отредактировать →
  отправить view-once → проверить, что бот прислал ожидаемое.

## Вне рамок (YAGNI)
- Группы, каналы, супергруппы.
- Веб-интерфейс / админка.
- Облачная БД, сервер/деплой (пока локально; сервер — отдельный этап позже).
- Несколько пользователей-получателей.
